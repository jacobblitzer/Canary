# Penumbra Workload — display-preset-driven auto-tests

## Purpose

Drive comprehensive Penumbra auto-tests by parameterizing over named
display presets rather than authoring per-knob Slop graphs. The user
asked for "the ability to load up presets so we can have a more
comprehensive autotest." This spec is the Canary-side contract that
makes that work.

## How it differs from existing workloads

`cpig` and `pigture` workloads run Grasshopper / Slop graphs that
produce viewport screenshots or Cycles renders. They drive Penumbra
indirectly (or not at all). The new `penumbra` workload runs Penumbra
Studio (or a headless equivalent in Phase 2) directly:

- Workload root: `Canary/workloads/penumbra/`.
- Fixtures (initial): a small set of named scene JSONs from
  `Penumbra/packages/studio/scenes/` (sdf-teapot, multi-field, atlas-blob).
- Tests: each test fixes (scene, displayPreset) and asserts a viewport
  snapshot.

## TestDefinition extension

Add an optional field to the existing `TestDefinition` schema (see
`spec/TESTS.md`):

```ts
interface SetupConfig {
  // existing fields …

  /**
   * Named Penumbra display preset to apply before the first checkpoint.
   * Resolved against:
   *   1. The workload's `display-presets/<name>.json` (workload-local).
   *   2. Penumbra's shipped catalog at `@penumbra/runtime/display-presets/<name>.json`.
   * If both exist, workload-local wins.
   *
   * When omitted, the renderer's default state is used (matches current behavior).
   */
  displayPreset?: string;
}
```

The harness:
1. After mounting `PenumbraPass` and loading the scene, resolves the
   preset by name (workload-local first, then runtime catalog).
2. Calls `pass.loadDisplayPreset(preset)`.
3. Logs the resolved preset path + checksum into the test run record.
4. Proceeds with checkpoints.

## display-presets directory

```
Canary/workloads/penumbra/
├── display-presets/
│   ├── default.json
│   ├── diagnostic-bricks.json
│   ├── diagnostic-aabbs.json
│   ├── particulate-cloud.json
│   ├── particulate-blend.json
│   └── monument.json
├── fixtures/
│   ├── sdf-teapot.scene.json
│   └── multi-field.scene.json
├── suites/
│   ├── display-matrix.json    # Cartesian preset × scene sweep
│   └── smoke.json             # one-test sanity check
└── tests/
    └── (auto-generated by `gen_display_matrix.py` from the suites)
```

A generator script (`scripts/gen_display_matrix.py`) reads the suites
and emits a test JSON per (scene × preset) pair, mirroring CPig's
existing `gen_retopo_slop_tests.py` pattern.

## Mode selection

Both `--mode pixel-diff` and `--mode vlm` are supported. Each preset has
a `vlmDescription` baked into the auto-generated test JSON, derived from
the preset's `description` field. The Cartesian sweep produces:
- N pixel-diff baselines for regression coverage.
- N VLM checkpoints for correctness coverage.

## Logging

Per the Slop-tap convention (`MultiVerse/CLAUDE.md` "Logging in Slop tests"
section), Penumbra workload tests should log:
- The resolved displayPreset name + path + checksum.
- The full DisplayState JSON at the moment of each checkpoint (via the
  Penumbra debug logger).
- WGSL compile errors (if any) — already captured by the existing logger.

There are no Slop graphs in this workload, so no `Log Tap` components.
The logger is the substitute.

## Cross-repo dependencies

| Repo     | Surface this workload depends on                                          |
|----------|---------------------------------------------------------------------------|
| Penumbra | `pass.loadDisplayPreset(json)` and the shipped preset catalog (ADR 0011). |
| Slop     | Not used.                                                                 |
| CPig     | Not used.                                                                 |

Per the cross-repo change protocol (`MultiVerse/CLAUDE.md`), changes to
the Penumbra preset catalog (adding/removing/renaming a preset) **must**
update this workload's `display-presets/` directory if the suite
references the affected preset, or the test will fail to load.

## Open questions

- **Headless vs Studio.** Phase 1 uses Penumbra Studio in a hidden
  browser window (Playwright/Puppeteer-driven, same harness as existing
  workloads). Phase 2 may add a true headless harness (`@penumbra/headless`
  ADR — currently a stub). Out of scope here.
- **Preset versioning.** Presets are JSON; today they live in HEAD of the
  Penumbra repo. If a baseline depends on a specific preset shape, it
  may break when the preset shape evolves. Mitigation: include the
  preset file's git SHA in the baseline filename, or freeze the preset
  schema with a `schemaVersion` field. Recommend `schemaVersion`.
